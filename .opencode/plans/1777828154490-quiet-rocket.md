# Deployment Plan: CUIDAR no servidor com Docker + Nginx + Cloudflare

## Objetivo

Instalar o CUIDAR no servidor via Docker, expor em `cuidar.tiagojct.eu` com proxy reverso Nginx e Cloudflare DNS.

## Pré-requisitos

- Servidor com Ubuntu/Debian
- Docker e Docker Compose instalados  
- Git instalado
- Cloudflare configurado no servidor (o opencode no servidor já tem as definições)
- SMTP credentials para envio de email

## Fluxo de Deploy

### 1. Clone do repositório

```bash
cd /opt
git clone https://github.com/tiagojct/esep-cuidar cuidar
cd cuidar
```

### 2. Configurar variáveis de ambiente

Criar ficheiro `.env` na raiz do projeto:

```bash
cat > .env << 'EOF'
SESSION_SECRET=<gerar-com-openssl-rand-hex-32>
ADMIN_EMAIL=admin@cuidar.tiagojct.eu
ADMIN_PASSWORD=<password-segura>
ADMIN_NAME=Administrador

# SMTP
SMTP_HOST=smtp.gmail.com
SMTP_PORT=465
SMTP_SECURE=true
SMTP_USER=<email>
SMTP_PASS=<senha-app>
SMTP_FROM=CUIDAR <tiagojacinto@med.up.pt>
EOF
```

### 3. Gerar SESSION_SECRET

```bash
openssl rand -hex 32
```

Usar o output no `.env` como valor de `SESSION_SECRET`.

### 4. Build e iniciar Docker

**Opção A - Build local:**
```bash
docker compose build
```

**Opção B - Via GHCR (se GitHub Actions já fez push):**
O `docker-compose.yml` já aponta para `ghcr.io/tiagojct/cuidar:latest`. Se o workflow GitHub estiver configurado para fazer push da imagem, basta:
```bash
docker compose pull
```

Iniciar:
```bash
docker compose up -d
```

Verificar logs:
```bash
docker compose logs -f
```

### 5. Configurar Nginx como proxy reverso

Criar configuração do site:

```bash
cat > /etc/nginx/sites-available/cuidar << 'EOF'
server {
    listen 80;
    server_name cuidar.tiagojct.eu;

    location / {
        proxy_pass http://localhost:3000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_cache_bypass $http_upgrade;
    }

    location /health {
        proxy_pass http://localhost:3000/health;
    }
}
EOF

# Ativar site
ln -sf /etc/nginx/sites-available/cuidar /etc/nginx/sites-enabled/

# Testar e recarregar
nginx -t && systemctl reload nginx
```

### 6. Configurar Cloudflare DNS

Adicionar registo A:
- **Nome**: cuidar
- **Alvo**: <IP do servidor>
- **Proxy**: Ativado (laranja)

### 7. Configurar SSL

Se o Cloudflare Proxy estiver ativado (laranja), o SSL é gerido automaticamente pelo Cloudflare.

Para forçar HTTPS no Nginx (recomendado), adicionar após configurar o certificado:

**Opção A - Cloudflare Full (strict):**
- Cloudflare Dashboard → SSL/TLS → Full (strict)
- O servidor precisa de um certificado válido (auto-assinado não funciona com strict)

**Opção B - Let's Encrypt:**
```bash
apt install certbot python3-certbot-nginx -y
certbot --nginx -d cuidar.tiagojct.eu
```

### 8. Verificar funcionamento

```bash
# Endpoint de saúde
curl http://localhost:3000/health
# Esperado: {"status":"ok","timestamp":"..."}

# Testar via domínio (se DNS já propagou)
curl https://cuidar.tiagojct.eu/health

# Página de login
curl -s https://cuidar.tiagojct.eu/login | grep -q "CUIDAR" && echo "Login OK"
```

### 9. Criar utilizadores de teste

Após deploy, aceder a https://cuidar.tiagojct.eu/admin/utilizadores e criar:

| Perfil | Email | Password |
|--------|-------|----------|
| Profissional de saúde | clinico@cuidar.local | clinico123 |
| Cuidador | cuidador@cuidar.local | cuidador123 |

**Nota**: O admin já é criado automaticamente via `ADMIN_EMAIL`/`ADMIN_PASSWORD`. As categorias, diagnósticos e fichas são semeados automaticamente pelo `production-setup.js`.

## Como funciona o setup automático

O `docker-entrypoint.sh` corre em cada start do container:
1. Executa `production-setup.js` que semeia categorias, diagnósticos e fichas
2. Cria o admin a partir das variáveis `ADMIN_EMAIL`/`ADMIN_PASSWORD`
3. Inicia a aplicação

O admin criado no primeiro start é **persistente** - se já existir, não é recriado.

## Ficheiros relevantes

| Ficheiro | Descrição |
|----------|-----------|
| `Dockerfile` | Build da imagem Node.js 22 Alpine |
| `docker-compose.yml` | Configuração do container com env vars |
| `docker-entrypoint.sh` | Script de inicialização (seeds + admin) |
| `.env` | Variáveis de ambiente (criar localmente) |
| `seeds/production-setup.js` | Seeds automáticos (categorias, diagnósticos, fichas) |
| `src/config.js` | Configuração lida de env vars |

## Verificação pós-deploy

```bash
# 1. Container está a correr
docker ps | grep cuidar

# 2. Logs sem erros
docker compose logs --tail=20

# 3. Health check responde
curl -s https://cuidar.tiagojct.eu/health | jq .

# 4. Página de login carrega
curl -s -o /dev/null -w "%{http_code}" https://cuidar.tiagojct.eu/login

# 5. Login funciona:
#    admin@cuidar.tiagojct.eu / <password definida>

# 6. Testar SMTP - enviar uma mensagem como cuidador
```
